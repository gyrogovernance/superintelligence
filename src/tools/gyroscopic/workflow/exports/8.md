# Export 8: `external/llama.cpp/ggml/src/ggml-gyroscopic/repack.cpp`

## Activation: when does repack `MUL_MAT` run?

1. Build: **`GGML_USE_CPU_REPACK`** registers `ggml_backend_cpu_repack_buffer_type()` in `ggml_backend_cpu_get_extra_buffer_types()` (`ggml-cpu.cpp`).
2. **Weight tensor** (typically `src0`) must live on that buffer: `op->src[0]->buffer->buft == ggml_backend_cpu_repack_buffer_type()`, 2D for `MUL_MAT`.
3. `ggml_repack_get_optimal_repack_type(op->src[0])` must be **non-null** (`supports_op` and `init_tensor` depend on this).
4. `src1->type == GGML_TYPE_F32` and host (or allowed) `src1` buffer (`supports_op` ~4773-4807).
5. Then `traits.cpp` calls `tensor_traits->compute_forward` -> **`forward_mul_mat`** / **`forward_mul_mat_id`** on the repack template class.

## `ggml_repack_get_optimal_repack_type` and **Q8_0 on x86**

For `cur->type == GGML_TYPE_Q8_0` (~4700-4721), the function returns a trait only under **Neon** (`ggml_cpu_has_neon()` + int8 mm or dotprod) or **RISC-V** branches.

There is **no** `ggml_cpu_has_avx2()` (or other x86) branch for `GGML_TYPE_Q8_0` in this selector.

**Consequence:** On a typical **Windows / x86 AVX2** build, **Q8_0** tensors on the repack buffer likely get **`nullptr`** optimal traits -> `init_tensor` sets `tensor->extra` from that null -> **`supports_op` fails** the `ggml_repack_get_optimal_repack_type(op->src[0])` check -> repack fast path for **Q8_0 MUL_MAT does not engage**; execution stays on **generic `ggml_compute_forward_mul_mat`** (gyro GEMM + vec_dot hooks from export 3).

**Caveat:** Q4 / Q4_K / etc. have AVX2 branches and **do** use repack on x86; this note is specific to **Q8_0 weight** repack selection.

## `forward_mul_mat` (~4254+)

- Asserts `src1` **F32**, 2D-ish batching, `ne03/ne13` constraints.
- Packs activations into **`PARAM_TYPE`** rows in `params->wdata` using `ggml_quantize_mat_t<INTER_SIZE, PARAM_TYPE>` (vectorized strips) and `from_float` for tail from `ggml_get_type_traits_cpu(PARAM_TYPE)->from_float`.
- Parallel chunking over `src0` rows / `src1` planes; **`forward_mul_mat_one_chunk`**:
  - If `nrows > 3`: **`gemm<BLOC_TYPE, INTER_SIZE, NB_COLS, PARAM_TYPE>(...)`**
  - Else tail: **`gemv<...>`** per row.
- For **`tensor_traits<block_q8_0, 4, 4, Q8_0>`** etc., `gemm`/`gemv` resolve to **`ggml_gemm_q8_0_*` / `ggml_gemv_*`** (declarations in `repack.h`; **generic reference** bodies in this file ~2281+; arch overrides on ARM/RISC-V).

## Layout (Q8_0 repack)

- Repack stores **interleaved / blocked** layouts (`block_q8_0x4` style) via `repack<BLOC_TYPE, INTER_SIZE, NB_COLS>(...)` in `tensor_traits::repack` (~4520-4523), not raw GGUF row-major Q8_0. Generic mul_mat sees **original** layout on **host** buffers; repack buffer holds transformed layout.

## Gyroscopic hooks

- **No `#ifdef GGML_USE_GYROSCOPIC`** in the repack forward path scanned. Repack GEMM/GEMV does **not** call `ggml_gyroscopic_gemm_q8_0_q8_0` or `ggml_gyroscopic_vec_dot_q8_0_q8_0`.

## Key answer (for Q8_0 on x86 desktop)

- **If** repack were active for Q8_0, matmul would **not** use generic `vec_dot`; it would use **repack `gemm`/`gemv`** (no gyro hooks today).
- **On current x86 AVX2 selector, Q8_0 optimal trait is null** -> repack MUL_MAT fast path for **Q8_0 weights** likely **off**; **generic path + hooks** apply for standard GGUF Q8_0 on host buffer.

## Gyroscopic action

- **Bypass risk:** If a future change adds **AVX2 Q8_0** entries to `ggml_repack_get_optimal_repack_type`, MUL_MAT could bypass vec_dot/GEMM hooks until repack kernels call the bridge.
- **log.md:** Record Q8_0 / x86 / repack selector gap and no-gyro-in-repack fact.
- **Kernel:** To own repack path, hook inside **`gemm<block_q8_0,...>`** call chain or arch `ggml_gemm_q8_0_*` implementations.
