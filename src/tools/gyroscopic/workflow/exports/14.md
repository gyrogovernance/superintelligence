# Export 14: `external/llama.cpp/tests/test-backend-ops.cpp`

## What it tests

Large GGML op suite: builds small graphs per **`test_case`**, runs **forward** (and optional **backward** / **perf**) on **multiple backends**, asserts **bitwise or tolerance-consistent** results across backends. Covers **`GGML_OP_MUL_MAT`**, **`GGML_OP_MUL_MAT_ID`**, fusions, KV types, many dtype combos.

## Q8_0 / matmul relevance

- Explicit **`test_mul_mat`**, **`test_mul_mat_id`**, and shape grids include **`GGML_TYPE_Q8_0`** with **`GGML_TYPE_F32`** activations (see registrations ~7982+ and loops ~8840+).
- Exercises **real backend dispatch** (CPU + any enabled GPU/RPC), not the gyro bridge in isolation.

## Q8_0 dot product

- Does **not** call `vec_dot` directly; it validates **whole-op outputs**. Failures implicate **mul_mat** stack (including **vec_dot** / **GEMM** inside CPU backend).

## Running against gyroscopic backend

- Upstream test binary links **`ggml`** targets chosen by CMake. To stress **ggml-gyroscopic**, the build must use the **gyroscopic CPU backend** for the CPU device (same as **`llama-cli`** build flags). If the test still loads a **vendor** CPU backend, gyro hooks never run.
- Passing after switching CPU backend: **gyro path matches reference backend** for covered ops (strong signal for numerical parity on those cases).

## Gyroscopic action

- **CI idea:** optional job: build tests with **`GGML_CPU_BACKEND_SUBDIR=ggml-gyroscopic`** + **`GGML_GYROSCOPIC=ON`**, run subset **`-o MUL_MAT`** (CLI filters documented ~9284+).
- **log.md:** Backend-ops parity is broader than **quantize-fns**; slower, higher confidence.
